01 · What holds up

The foundation is honest

Before the gaps: most of what the documents call sacred is genuinely held in the build.

  • The privacy boundary is honored everywhere. The Nudge renders as a private overlay with "Seen only by you. Nothing public, no announcement." The Command Center's nudge queue carries "Each nudge goes to one recipient only — never to a shared view." No shared surface ever shows a match suggestion. This is the highest-stakes rule in the app and it passes everywhere.
  • Color semantics are mostly disciplined. Green appears on interactive and confirming elements; purple holds its "only for you" meaning across the Nudge, follow-up signals, and held containers; amber carries the Chance countdown; red appears only on the failed code, the cancel path, and the cancelled state.
  • The brand-agnostic rule reads correctly. Attendee screens wear Futureland (typed name, no logo, graceful fallback); SODA stays a quiet footer credit; the Admin's logo field literally says "No logo — host name shown in type."
  • The voice survives the build. "You made it in." "Ninety seconds to build your card." "Nothing to review. You are caught up." Copy across the kits matches the docs' warm, plain register, and several states-doc lines are used verbatim.
  • HCD basics are real, not decorative: photo skip never blocks (initials fallback), sign-in error is recoverable with resend, chip caps soft-lock instead of erroring, end-of-night destructive action is two-step confirmed with a graceful default.
  • The data-ownership clause shows up in UI: the Export panel is titled "Futureland owns its data" — the contract made visible.
02 · P1 — Spec breaks

Rules the documents state that the build does not enforce

2.1 · Identity fields never lock P1

The Event Layer is explicit: "Identity is editable in draft, locked in live, and final in closed." The built Admin shows the rule in the lifecycle seam copy — but the Event name, Host name, and Date inputs stay fully editable in every phase. Flipping to Live or Closed changes the banner, not the fields. The one behavior the lifecycle exists to protect (a stable identity while guests are inside the experience) isn't enforced.

Fix: pass phase into EventSetup; disable identity inputs when not draft, with a quiet mono line — "Locked while live" / "Final — the event is a record."

2.2 · The Command Center can fire acts in any state P1

Lifecycle State Awareness (inventory #38) is the gate that "protects the night from misfires": act controls active only when live, visibly inert otherwise. The built cockpit hardcodes a Live badge and every control is always armed. There is no draft or closed rendering at all, and no way to see what "visibly inert" looks like.

Fix: give the Command Center a phase (read from the same state the Admin sets), render Fire/Spin/Send disabled with the existing 0.5-opacity disabled treatment plus a one-line reason when not live.

2.3 · The lifecycle message has one of its two variations P1

The Event Layer defines "a single flexible screen with two text variations" — not-yet-open and event-ended (the ended one carrying a path to the survey). The attendee spine builds only "Doors aren't open yet." A guest scanning a closed event has no designed destination, and the closed state is the one that feeds the survey loop.

Fix: add the ended variation to the existing Lifecycle screen — headline, one warm line, and a green "Take the survey" path when uncompleted.

2.4 · Sign-in collects data the privacy policy doesn't cover P1 · decision needed

Every document specifies sign-in as email + six-digit code. The built attendee sign-in adds a phone-number path (default!) and a "Continue with Google" option. The privacy policy states "Your name and email address, entered at sign-in… That is the full list." Phone numbers and a Google identity provider are both outside that list, and Google sign-in also bypasses the "identity is established once, counts once toward MAU" model the inventory describes.

Fix (either direction): if phone/Google are intentional product evolution, update the privacy policy, flow map, and inventory to match. If not, strip sign-in back to email + code. The build and the legal document cannot disagree.

03 · P2 — The states debt

The moments the states doc says happen every event

The states pass names five states to "build first, these happen every time." Two are built. The doc's own rule — "the first guest and the no-match guest are not edge cases" — is currently unmet.

Build-first stateStatusWhat exists
The early roomMissingRoomView always renders 12 people. No climbing count, no dimmed placeholders, no "The room is filling. You are early. Good." — called the most important empty state in the app.
The Drop waiting wallPartialThe spec copy lives in the input placeholder — a nice touch — but after sending, the wall is instantly full of seeded answers. The empty-wall beat ("yours can be first") never renders.
The Nudge no-matchMissingThe attendee Nudge always has a match. No purple-treatment fallback card, despite a paste-ready prompt for it.
CC / Admin zero states2 of 8Chip queue ("Nothing to review. You are caught up." — verbatim ✓) and nudge queue have empties. Stat bar, room, feed, sync, chance, survey, and intel always render full.
Sign-in errorsBuiltWrong code → red line with spec copy, field ready to retry, resend offered. Send shows "Sending your code…" locked. Matches the doc.

Lower-priority states, for the record: survey submitting/failed missing (Finish advances instantly; the answers-preserved retry isn't designed) · photo uploading/failed missing (though skip→initials cushions it) · room loading skeleton, reconnecting indicator, connection-badge error missing (badge is permanently "Synced") · Chance unpaired/shuffling missing · export empty missing (export loading built — "Preparing…", control locked) · thin-history recognition missing (Recognition always claims "third event, seven connections").

Recommendation: a "States" toggle in each kit's Tweaks panel (empty / loading / error / populated) would make every state designable and reviewable without forking screens — and this project already has the Tweaks pattern wired in the Admin.

04 · P2 — Inventory drift

"Built" in the doc, absent in the system

The Component Inventory grounds its statuses in the original prototype screens. Six host-side components it marks Built have no home in this design system's kits — the place a consuming project would actually look:

Component (inventory #)ClaimedIn this system

Question Creator (#33)BuiltAbsent.
Open Mic Feed (#34)BuiltAbsent.
Broadcast & Announce (#35)BuiltAbsent — no way to message the whole room from either tool.
Room Panel, Admin (#30)BuiltAbsent — the room grid lives only in the Command Center.
Survey Panel, Admin (#31)BuiltAbsent — survey data lives only in the CC monitor; the host flow map expects the fuller Admin table for the closed-state review.

This matters because the Host Flow Map's Part C — "Back in the Admin" (review matches and survey data, then export) can't be walked in the built Admin: matches and export exist, the survey review does not.

Fix (either direction): port these six panels into the Admin kit, or restate their inventory status as "Built in legacy prototype, not yet in the design system." A status table that can't be trusted decays fast.

06 · The Home

Home spec compliance

The strongest spec-to-build match in the project. Shell, greeting, recent-events strip, synced-people block, deep-green insight card, dismissible follow-ups overview with the purple "People to reach out to" group, search, signals, and the purple tab dot are all present and correctly toned. Gaps:

  • Reserved affordances were dropped P2. The spec deliberately reserves a scan-to-add icon in the Contacts header and a share/quick action on each contact row so the "soon" features slot in additively. Neither exists, so those features will require layout change later — exactly what the reservation was for.
  • Event rows lack the recap/survey affordance P3: "a recap or survey affordance appears where one is available."
  • Met-at line format P3: built rows read "Investor · Creative Meetup, Cleveland"; the spec's pattern is "Met at Creative Meetup · Cleveland." Minor, but the "Met at" framing is doing memory work.
  • Rows aren't semantic P2 · a11y: contact and event rows are clickable divs — no keyboard path, no focus state. Same applies to the room's person cards.

7.3 · Type scale and the 8px grid

  • 9–10px mono is everywhere — section labels, signal pills, met-counts, footers. On a real 380px phone these are sub-7pt. The docs' own floor for print is 12pt; nothing is stated for screen, but a stated 11px minimum for mono labels (and reserving 9px for the device mock only) would be worth adding to the type guideline. P3
  • The "8px grid, 16px margins" claim is loose in practice: paddings of 7, 9, 11, 13, 15, 17px are routine across the kits. The rhythm still reads consistent because it's consistently off — but the docs repeat "8px grid" in every prompt. Either soften the doc (a 4px base grid describes reality better) or sweep the kits. P3
